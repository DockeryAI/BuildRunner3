# Second Opinion Brief — Round 3: validate the converged plain-English plan

**Goal:** After two prior Codex rounds (initial RCA → flock; revised RCA → state-machine fix + queue stub promotion), the plan converged. Operator asked for the plan in plain English. Validate the converged plan as actually-shippable: any remaining blockers, missed cases, sequencing problems, or wrong-shape choices.

**Question:** Is this six-step plain-English plan complete and in the right order? Specifically:
(a) Does fixing the state-machine bug FIRST (before recovery and before the queue) actually solve the immediate cohort, or does the cohort recovery need a different mechanism that doesn't depend on the state fix?
(b) Is "single worker inside watchd" the right home for the queue worker, or does watchd's fswatch/reaper duty fight with serialized fix dispatch (e.g. a 30-min fix attempt blocks all watchd reactivity)?
(c) Does "skip path immediately marks needs_human" lose any signal we'd want (e.g. operator wants to flip autoheal back on and resume — does needs_human kill that affordance)?
(d) Is the cost-cap accounting fix actually necessary in this plan, or is it a separate concern that only matters once the queue is live?

## Converged Plain-English Plan (what was delivered to operator)

**The core problem.** A batch of cases got marked "being fixed" but auto-heal was switched off at the time, so nothing ever fixed them. They sat there until a janitor process eventually relabeled most of them "needs a human." Underneath that, there's a deeper bug: even when auto-heal IS on and a fix actually works, the code forgets to move the case out of the "being fixed" bucket — so cases would get stuck again the next time around.

**What we're going to do, in order:**

1. **Fix the bookkeeping bug first.** When a fix attempt succeeds, the case should go back into the probation lane and start earning its way back to "promoted." Right now it just silently stays in "being fixed" forever. This is the single most important change.

2. **Give stranded cases a proper recovery path.** The existing "fix this one" command wasn't designed to pick up cases already written off as "needs a human" — pointing it at them quietly does the wrong thing. Add a real recovery command that knows how to pull a case back into the active lanes before retrying.

3. **Build one queue with one worker.** Today, anything that triggers a fix runs it right then and there in whatever process noticed the failure — operator, file-watcher, canary, all independently. Change that so every trigger drops a ticket into a queue, and a single worker (living inside the existing watcher daemon) pulls one ticket at a time and runs it. No two fixes ever run at once. The lock is the queue itself, not a last-second mutex.

4. **Stop the silent stranding.** When auto-heal is off and a case fails, don't leave it labeled "being fixed" — immediately label it "needs a human, auto-heal was off" so the state of the system tells the truth.

5. **Make status commands read-only.** Right now, just checking status can trigger the janitor to start relabeling things. Move all cleanup work into the worker and the watcher.

6. **Fix the cost-cap accounting.** The daily spending cap currently counts every queue ticket as if it were a real fix attempt. Once we have a real queue minting tickets, that math breaks. Count actual attempts instead.

## Technical Backing for Each Step (from prior briefs)

### Step 1 — state-machine bug (the keystone)

`state_advance_after_run` (state.sh:84-94) and `db_advance_and_transition` (db.sh:286-291) both have the same bug: when src=="fixing" and passed==1, they do not transition to probation. They return src unchanged. The transition `fixing → probation` is in the allowed table at state.sh:26-33 but no code calls it.

Two divergent state-machine implementations exist. The hot path is `db_advance_and_transition` which writes raw SQL bypassing `state_transition_allowed`. Step 1 should fix BOTH (or consolidate to one).

### Step 2 — recovery path for needs_human

The 36 reaped cases are in `needs_human`. `cmd_fix` (burnin.sh:296) skips state checks and calls `fix_loop_run` → `runner_run` → `db_advance_and_transition`. That function has no case for src=="needs_human" — it falls through with target=src and emits no transition SQL, while the `requested` stub still gets inserted by runner.sh:185-204. Result: silent half-state.

Recovery command should explicitly transition `needs_human → untested` (allowed by table) before invoking fix loop, OR enqueue with an explicit "recovery" intent that the worker honors.

### Step 3 — queue + single worker

Today: operator + watchd + canary + manual all call `fix_loop_run` in-process. Worktree branch isolation only protects same-case-same-attempt. Cross-process collisions are possible.

Proposal: promote `fix_requests.status='requested'` into a real queue. Worker claims oldest with `BEGIN IMMEDIATE; SELECT … LIMIT 1; UPDATE … SET status='proposed', worker_id=?, claim_at=now COMMIT`. Triggers enqueue only — never dispatch. Single consumer = single LaunchAgent (watchd) running the worker as a sidecar tick.

### Step 4 — skip path

burnin.sh:271-288: when AUTOHEAL=off, currently records `db_record_fix_skip` but leaves state at `fixing`. The reaper later flips to `needs_human` after 30 min. Step 4 says: skip path itself should immediately transition `fixing → needs_human` with reason "autoheal off" — no waiting on reaper, no rotting in fixing.

### Step 5 — read-only status

`cmd_status` (burnin.sh:81-119) calls `state_reap_stale_fixing`. Reading shouldn't mutate. Move reap into the worker/watchd loop only.

### Step 6 — cost cap

`cost_check_daily_cap` (cost.sh:166) counts ALL fix_requests rows. Once the queue mints a `requested` row per failure, the cap will trip on enqueue volume not actual claude spend. Should count rows where claude was actually invoked (status='proposed' or terminal with attempts>0).

## Doubts to Stress-Test

- **Doubt A:** Is sequencing right? Step 1 (state bug) must land before Step 2 (recovery) — otherwise a successful recovery doesn't transition correctly. But should Step 3 (queue) actually go before Step 2, since Step 2 will dispatch fixes that should already be queue-routed?

- **Doubt B:** Step 3 puts the worker inside watchd. If a fix takes 15 minutes (3 attempts × 300s timeout each = 900s = 15 min, plus overhead), watchd's fswatch event loop is blocked for that whole window. Should the worker be a separate LaunchAgent that shares a SQLite advisory lock with watchd's reaper, instead of being co-located?

- **Doubt C:** Step 4 immediately marks needs_human on autoheal=off. But the operator's typical flow is "I turned autoheal off to do a sweep, now I'm turning it back on, please pick up where you left off." If those cases are in needs_human, they require explicit recovery (Step 2) — which is more friction than the current implicit "rerun and they'll re-enter fixing" behavior. Is the right design to use a NEW state like `deferred_fix` that's allowed to transition back to `fixing` directly when autoheal flips on?

- **Doubt D:** Step 6 changes cap accounting. But the queue might never enqueue more than a handful of `requested` rows at a time if dedup is in place (one open request per case). Is the cap fix actually load-bearing or premature?

- **Doubt E:** The plan never addresses the runner.sh atomicity gap (db_advance_and_transition + db_record_fix_request_stub are two separate sqlite calls). With the queue, this becomes "transition + enqueue must be atomic, otherwise a crash between them strands the case." Should this be Step 2.5 / Step 3 prereq?

- **Doubt F:** The plan says "single worker." What's the recovery story if the worker process dies mid-attempt holding a `proposed` claim? Reaper currently uses created_at not claim_at. Plan implies fixing reaper to use claim_at + heartbeat — is that explicitly Step 3 scope or implicit?

- **Doubt G:** No mention of the `cmd_fix` operator escape hatch. If operator runs `burnin fix <id>` after Step 3 lands, does it bypass the queue or does it enqueue and wait? Both have ergonomic costs.

## What success looks like

- No case ever stuck in `fixing` without an active worker
- A successful fix-loop attempt provably moves the case out of `fixing`
- All fix dispatches serialize through one queue, one worker, machine-wide
- Operator-visible read commands have zero side effects
- Operator can recover stranded `needs_human` cohort with one explicit command
