## Claude's Revised Plan (after your last review)

1. **Fix the state-machine pass path first.** Add `fixing → probation` on green in both `state_advance_after_run` (state.sh:84-94) and `db_advance_and_transition` (db.sh:286-291). On a green re-run from `fixing`, transition to `probation` so the case can resume earning consecutive greens toward `promoted`.

2. **Recover the cohort as `needs_human`, not `fixing`.** Iterate the 36 reaped cases and run `burnin fix <id>` serially in a one-line bash loop. (Concern: state-machine table at state.sh:26-33 says `needs_human → probation | untested` — does `burnin fix` actually run from `needs_human`? `cmd_fix` at burnin.sh:296 doesn't check state, just calls `fix_loop_run` which calls `runner_run` which calls `db_advance_and_transition`. If the run still fails, db_advance_and_transition will try `fixing` from `needs_human` — not in the allowed table. Does it silently fail?)

3. **Close the runner.sh atomicity gap.** Move `db_record_fix_request_stub` into the same `BEGIN IMMEDIATE; … COMMIT;` block as `db_advance_and_transition`'s state flip, instead of two separate sqlite calls.

4. **Replace in-process dispatch with a single dequeue worker.** Promote the existing `fix_requests.status='requested'` stub into a real work queue. One consumer (own LaunchAgent or watchd-internal worker) selects the oldest `requested` row, claims it (status `proposed` with a worker-id and claim_at), then runs `fix_loop_run`. Operator/watchd/canary/manual all enqueue (i.e. enter `fixing` state via the existing path which already writes the stub) but never dispatch in-process. Remove the in-process `fix_loop_run` call from `_run_one`.

5. **Leave `state_reap_stale_fixing` as cleanup only.** No mutation to dispatch fixes from the reaper.

## Specific Doubts Worth Stress-Testing

- **Doubt A:** Should the green-from-fixing transition go to `probation` (resume the qualifying ladder) or directly to `promoted` if `consecutive_greens >= promote_after` is already satisfied? The `db.sh:286-291` block currently only awards `promoted` when src is `probation`. If we add `fixing → probation` on green AND a fix-loop case had accumulated greens (it can't right now, but if the bug is fixed it could), we should consider the promotion gate too.

- **Doubt B:** The dequeue worker design (item 4) breaks the operator's manual `burnin fix <id>` flow — that command bypasses the queue. Should `cmd_fix` enqueue and wait, or remain a synchronous escape hatch?

- **Doubt C:** `_run_one` has SIGINT trap behavior that records a fix_skip. If we remove in-process dispatch, do we lose the SIGINT-records-skip property for operator-interrupt cases?

- **Doubt D:** The `db_advance_and_transition` function emits SQL inside a single transaction. Inserting `fix_request_stub` SQL into the same `db_exec` call (item 3) requires either passing the run_dir into `db_advance_and_transition` (signature change, called from runner.sh and one other place) or building the SQL in runner.sh and using a single `db_exec` from there. Which is right?

- **Doubt E:** Watchd is currently configured as a singleton LaunchAgent. If we add a separate dequeue-worker LaunchAgent, do they collide on the SQLite write lock? Should the worker be inside watchd?

## What success looks like

- Stuck cases never accumulate again. `BR3_BURNIN_AUTOHEAL=off` sweeps end with cases in a state that names the operator action (e.g. `needs_human` with reason "autoheal off — re-enable to retry"), not silently in `fixing`.
- A green fix-loop attempt actually advances state out of `fixing`.
- Every fix dispatch goes through one queue. No two `fix_loop_run` calls execute concurrently across processes.
- Operator-visible commands (`burnin status`, `burnin list`, `burnin show`) never trigger work as a side effect.
