## Claude's Current Hypothesis (Root Cause + Fix Plan)

**Root Cause:** Phase 4 of the autopilot build (`burnin run --untested`) executed with `BR3_BURNIN_AUTOHEAL=off` deliberately set. The autoheal branch in `_run_one` (burnin.sh:271–288) detected the off-flag and aborted every dispatch, recording `aborted/skipped` rows with reason `"autoheal disabled (BR3_BURNIN_AUTOHEAL=off)"` instead of invoking `fix_loop_run`. The cases now sit in `fixing` indefinitely because:

- No autoheal will ever fire for them (the trigger has already passed and the dispatcher is one-shot per failure).
- The 30m reaper at `state.sh:137-175` only re-states them to `needs_human`, it does not retry a fix.
- Watchd only re-runs cases on file-change events that match a case's `watch_paths`; no event has matched any of these 36 cases since the sweep.

**Concurrency answer:** Within a single `burnin run` invocation the harness is already serial (`_run_untested`/`_run_matching` use a `for`/`while read` loop and call `_run_one` synchronously; `_run_one` calls `fix_loop_run` inline before iterating). What is _missing_ is a cross-invocation lock — if operator + watchd + canary all fire simultaneously, fix-loops for different cases can run concurrently. Worktree branch `burnin-fix/<case_id>-<attempt_n>` only protects same-case-same-attempt collisions.

**Fix Plan:**

1. Re-enable autoheal for the session (`export BR3_BURNIN_AUTOHEAL=on`), reset the 36 cases to `untested` (`burnin reset` or bulk SQL `UPDATE burnin_cases SET state='untested', fix_attempts=0, consecutive_greens=0 WHERE state='fixing'`), then `burnin run --untested` to re-drive failures through `probation → fixing` with autoheal live.
2. Add a global fix-dispatch lock in `fix_loop_run` (fix-loop.sh:153) using `flock` on `~/.buildrunner/burnin/fix-dispatch.lock`, honoring `BR3_BURNIN_FIX_LOCK_WAIT_S` so concurrent callers queue rather than race.
3. Change the no-autoheal sweep contract: when `_run_untested` runs with autoheal off and a case fails, finalize it to `needs_human` (or a new `deferred_fix` state) instead of leaving it in `fixing`, so state truthfully reflects "waiting on human" rather than rotting.
4. Make `state_reap_stale_fixing` smarter: when the only fix_requests row is `aborted/skipped` with `BR3_BURNIN_AUTOHEAL=off` reason and autoheal is now on, re-queue the case for a real fix dispatch (subject to the lock from step 2) instead of unconditionally flipping to `needs_human`.
