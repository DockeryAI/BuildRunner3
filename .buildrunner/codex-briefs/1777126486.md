# Second Opinion Brief — /research pipeline triple-failure root cause

**Goal:** Confirm or contradict three independent root-cause findings from a /research pipeline failure analysis. The pipeline shipped two parallel research runs on the same topic (Neolithic henge-builders); one run committed to git but with a hidden reindex warning, the other failed entirely with a timeout, and Jimmy's `/retrieve` returns ~0.0076 (noise) when queried — meaning the LanceDB index never actually picked up either doc.

**Question:** Are the three root causes below correctly identified? Are the proposed fixes adequate, or is there a deeper issue any of them is masking?

## Claude's Current Hypothesis

### Root Cause 1 — Perplexity returned credit-union content

The phrase "common misconceptions" embedded in the output-format placeholder of the sub-agent prompt template causes sonar-pro's grounded web search to retrieve credit-union "common misconceptions" FAQ pages (the dominant traffic for that phrase) instead of archaeology content. Compounded by the absence of any `search_domain_filter`. Fix: strip `<output_format>` block from non-Claude prompts; add domain filter to perplexity.py; replace placeholder text "common misconceptions" with neutral "evidence gaps and caveats".

### Root Cause 2 — `--body-file` rejected

Dispatcher (committed 2026-04-22 16:08:39) implements positional-provider API; skill's reviewer (committed 2026-04-22 16:25:58, 17 min later) calls flag-first API with `--provider --role --body-file`. `--provider` lands as positional `$1`, sets `PROVIDER="--provider"`, fails allowlist, exits 3 with `invalid_provider:--provider`. Even if --provider passed, --body-file isn't recognized — silently logged to stderr (which goes to .err files the skill never reads) and shifted away. Codex receives empty/placeholder body. All three reviewer providers fail identically, cascade to `degraded_no_review`. Fix: update dispatcher to accept `--provider/--role/--body-file` as flags; update Python clients to read body-file; make wildcard handler exit nonzero with structured JSON error.

### Root Cause 3 — LanceDB chunk_count timeout

Two compounding faults: (a) 60s timeout < cold-path indexing latency (sentence-transformers cold load + embed + count_rows over 20,470 rows can exceed 60s; the existing 300s background interval implies this pipeline takes minutes); (b) silent "already indexing" race — if the 5-min background loop is mid-run when the POST arrives, Jimmy returns `status: already_indexing` and the new thread exits immediately at line 253; worker doesn't read the response, polls a counter that's frozen until the prior job finishes. Worker writes `status: ok` because timeout returns a tuple instead of raising. Fix: raise timeout to 180s; parse POST response and handle `already_indexing` by waiting on `last_index > commit_time_epoch` instead of chunk_count delta; add `wait=true` synchronous-with-timeout option to Jimmy's reindex endpoint; treat reindex timeout as `status: indexing_pending` (not ok); add post-commit `/retrieve` verification before declaring done.

### Common thread

All three failures share one pattern: a downstream component reports success on a partial result, and the upstream caller doesn't verify the actual outcome. The structural fix is end-to-end verification — queue isn't "done" until a `/retrieve` query for the committed doc returns it with a real score.
